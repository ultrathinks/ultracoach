# 기술 면접 준비 전략

기술 면접은 단순히 "코딩을 잘하는지" 확인하는 자리가 아닙니다.
**문제를 어떻게 분석하고, 어떻게 접근하며, 어떻게 소통하는지**를 종합적으로 평가합니다.
기술 면접의 유형별 준비 전략과 실전 팁을 정리했습니다.

---

## 기술 면접 유형

### 1. 코딩 테스트 (알고리즘)

온라인 또는 현장에서 알고리즘 문제를 푸는 형식입니다.

**평가 요소:**
- 문제 이해력과 분석 능력
- 자료구조/알고리즘 선택의 적절성
- 코드 구현 정확도
- 시간/공간 복잡도 분석
- 엣지 케이스 처리

**준비 전략:**
- 자주 출제되는 패턴 15개를 집중 연습하세요
- 해시맵, 투 포인터, 슬라이딩 윈도우, BFS/DFS, DP가 핵심
- 하루 2문제씩 4주면 기본기가 잡힙니다
- 문제를 푸는 것보다 **설명하면서 푸는 연습**이 더 중요

**실전 팁:**
- 문제를 받으면 바로 코딩하지 마세요
- 1~2분 간 **질문을 통해 조건을 명확히** 하세요
- 접근 방법을 먼저 말로 설명한 후 합의를 얻고 코딩 시작
- "이 방법의 시간 복잡도는 O(n)이고..." 식으로 분석을 먼저 공유

### 2. 시스템 설계 (System Design)

대규모 시스템의 아키텍처를 설계하는 면접입니다. 보통 시니어 레벨부터 출제됩니다.

**평가 요소:**
- 요구사항 분석 능력
- 확장성(Scalability) 고려
- 트레이드오프 분석 능력
- 기술적 깊이와 폭

**준비 전략:**
- 자주 나오는 시스템 10개를 정리하세요:
  - URL 단축기, 채팅 시스템, 뉴스 피드, 검색 엔진
  - 알림 시스템, 파일 저장소, 동영상 스트리밍
  - 실시간 리더보드, 예약 시스템, 결제 시스템
- 각 시스템의 **핵심 컴포넌트와 데이터 흐름**을 이해하세요
- "System Design Interview" 서적으로 프레임워크를 학습하세요

**실전 팁:**
- 처음 5분은 **요구사항 질문**에 투자하세요
  - DAU는? 읽기/쓰기 비율은? 지연 시간 요구는?
- 상위 설계(High-Level Design)를 먼저 그리고 승인받으세요
- 세부 설계로 들어갈 때 **가장 중요한 컴포넌트**부터
- 모르는 부분은 솔직히 말하고, 아는 부분에서 깊이를 보여주세요

### 3. 기술 경험 면접

과거 프로젝트에서의 기술적 의사결정과 경험을 질문합니다.

**평가 요소:**
- 기술적 의사결정 능력
- 문제 해결 과정
- 기술적 깊이
- 협업과 커뮤니케이션

**자주 나오는 질문:**
- "가장 어려웠던 기술적 문제와 해결 방법"
- "기술 스택 선택 이유"
- "성능 최적화 경험"
- "장애 대응 경험"
- "코드 리뷰에서 받은 피드백"

**준비 전략:**
- 3~5개 프로젝트의 기술적 결정을 STAR로 정리하세요
- 각 결정의 **트레이드오프**를 명확히 설명할 수 있어야 합니다
- "왜 A가 아니라 B를 선택했는가"에 답할 수 있어야 합니다

### 4. 라이브 코딩

면접관 앞에서 실시간으로 코딩하는 형식입니다.

**평가 요소:**
- 코딩 실력 (당연히)
- **사고 과정 공유** (Think-aloud)
- 피드백 수용력
- 디버깅 능력

**실전 팁:**
- **말하면서 코딩**하세요 — 침묵 속 코딩은 면접관이 평가할 수 없음
- "여기서 null 체크를 해야 할 것 같은데..." 식으로 사고 과정 공유
- 막히면 "여기서 고민되는 부분은..."이라고 솔직히 말하세요
- 완벽한 코드보다 **동작하는 코드를 먼저**, 그 후 개선

## 기술 면접 공통 전략

### 질문하기

기술 면접에서 질문은 **감점이 아니라 가점**입니다.

좋은 질문 예시:
- "입력 데이터의 크기 제한이 있나요?"
- "동시 접속자 수는 어느 정도를 가정하면 될까요?"
- "정렬된 데이터인가요?"
- "메모리 제약이 있나요?"

이런 질문은 실무에서 요구사항을 명확히 하는 능력을 보여줍니다.

### 트레이드오프 사고

기술 면접의 핵심은 **정답을 아는 것이 아니라 트레이드오프를 분석하는 것**입니다.

항상 이 프레임으로 생각하세요:
- "A 방법은 ~한 장점이 있지만 ~한 단점이 있습니다"
- "B 방법은 ~한 상황에서 더 적합합니다"
- "이 요구사항에서는 ~를 우선시해야 하므로 A를 선택하겠습니다"

### 모르는 것 인정하기

모르는 질문이 나올 때:

**나쁜 대응:**
> "음... 그건... 어... (긴 침묵)"

**좋은 대응:**
> "그 기술은 직접 사용해본 적은 없지만, 제가 아는 범위에서 말씀드리면 ~와 비슷한 개념으로 이해하고 있습니다. 만약 이 프로젝트에서 필요하다면 ~한 방법으로 학습하겠습니다."

솔직함 + 학습 의지가 면접관에게 가장 좋은 인상을 줍니다.

## 유형별 시간 배분

### 코딩 테스트 (45분 기준)

| 단계 | 시간 | 활동 |
|------|------|------|
| 이해 | 5분 | 문제 읽기, 조건 질문 |
| 설계 | 5분 | 접근 방법 설명, 합의 |
| 구현 | 25분 | 코딩 + 사고 과정 공유 |
| 검증 | 5분 | 테스트 케이스 검증 |
| 최적화 | 5분 | 개선점 논의 |

### 시스템 설계 (45분 기준)

| 단계 | 시간 | 활동 |
|------|------|------|
| 요구사항 | 5분 | 기능/비기능 요구사항 질문 |
| 추정 | 5분 | 트래픽, 저장 공간 계산 |
| 상위 설계 | 10분 | 주요 컴포넌트 + 데이터 흐름 |
| 세부 설계 | 20분 | 핵심 컴포넌트 깊이 다루기 |
| 마무리 | 5분 | 확장성, 모니터링, 장애 대응 |

## 기술 면접 체크리스트

면접 전 이것만 확인하세요:

### 알고리즘
- [ ] 자주 쓰는 자료구조 5개 구현 가능한가?
- [ ] 핵심 알고리즘 패턴 10개를 설명할 수 있는가?
- [ ] 시간/공간 복잡도를 바로 분석할 수 있는가?

### 시스템 설계
- [ ] 대표 시스템 5개의 아키텍처를 그릴 수 있는가?
- [ ] 캐싱, 로드밸런싱, 샤딩을 설명할 수 있는가?
- [ ] CAP 정리와 트레이드오프를 이해하는가?

### 기술 경험
- [ ] 3개 이상 프로젝트의 기술 결정을 STAR로 정리했는가?
- [ ] 각 프로젝트의 아키텍처를 화이트보드에 그릴 수 있는가?
- [ ] 실패/장애 경험과 대응을 정리했는가?

## UltraCoach 활용

UltraCoach의 기술 면접 모드에서:

- **기술 경험 질문**을 반복 연습하세요
- STAR 충족률로 답변 구조를 점검하세요
- 추임새 히트맵으로 기술 설명 시의 유창성을 확인하세요
- 같은 프로젝트 경험을 3번 반복하면 설명이 간결해지고 핵심만 남습니다

기술 면접은 지식보다 **커뮤니케이션**에서 당락이 갈리는 경우가 많습니다.
알고 있는 것을 **정확하고 간결하게 전달**하는 연습이 가장 중요합니다.
