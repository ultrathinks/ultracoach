In `src/widgets/nav/nav-bar.tsx`, the active state logic at line 93 currently uses exact match:

```ts

```

Change the className logic in the links.map callback (around line 91-96) to use `startsWith` for the dashboard route while keeping exact match for other routes:

```ts
pathname === link.href || (link.href === "/dashboard" && pathname.startsWith("/dashboard"))
```

```tsx
className={cn(
  "text-sm transition-colors",
  pathname === link.href ||

    : "text-muted hover:text-secondary",
)}
```
</action>

<task id="11-02-04">
<title>Add signIn callbackUrl to redirect to /dashboard after login</title>
<read_first>
- /Users/dgsw67/ultracoach/src/widgets/nav/nav-bar.tsx
- /Users/dgsw67/ultracoach/src/shared/lib/auth.config.ts
</read_first>
<action>
In `src/widgets/nav/nav-bar.tsx`, change the signIn call (line 107) from:

```ts
onClick={() => signIn("google")}
```

to:

```ts
onClick={() => signIn("google", { callbackUrl: "/dashboard" })}
```

This ensures that after Google OAuth login completes, the user is redirected to /dashboard.

Note: Adding a `redirect` callback to `auth.config.ts` is an alternative but modifying the signIn call is simpler and doesn't risk breaking Edge compatibility. The `callbackUrl` approach is the recommended NextAuth pattern for single-callsite redirects.
</action>
<acceptance_criteria>
- `grep 'callbackUrl: "/dashboard"' src/widgets/nav/nav-bar.tsx` matches
- `grep 'signIn("google", { callbackUrl: "/dashboard" })' src/widgets/nav/nav-bar.tsx` matches
</acceptance_criteria>
</task>
