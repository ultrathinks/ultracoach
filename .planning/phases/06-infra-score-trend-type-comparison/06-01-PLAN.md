These types represent the analytics data contract between `features/analytics/compute-analytics.ts` and all `widgets/history/*.tsx` chart components. No runtime code -- pure type definitions and re-exports.

Create `src/entities/analytics/types.ts` with these exact interfaces:

```typescript
export interface ScoreTrendPoint {
  sessionId: string;
  createdAt: string; // ISO 8601
  deliveryScore: number | null;
  contentScore: number | null;
  interviewType: string;
}

export interface TypeComparisonGroup {
  type: string;        // "personality" | "technical" | "culture-fit"
  typeLabel: string;   // "인성" | "기술" | "컬처핏"
  avgDelivery: number; // 0-100
  avgContent: number;  // 0-100
  count: number;
}

export interface ChangeRate {
  deliveryChange: number; // percentage: (latest - first) / first * 100
  contentChange: number;
  hasEnoughData: boolean; // false when sessions < 2
}

export interface DashboardStats {
  totalSessions: number;
  recentWeekSessions: number; // sessions in last 7 days
  changeRate: ChangeRate;
}

export interface DashboardAnalytics {
  scoreTrends: ScoreTrendPoint[];
  typeComparison: TypeComparisonGroup[];
  stats: DashboardStats;
}
```

Key design notes:
- `ScoreTrendPoint` includes `interviewType` for potential future filtering, but the LineChart uses all points
- `TypeComparisonGroup` includes pre-computed `typeLabel` so chart components do not need the label map
- `ChangeRate.hasEnoughData` explicitly signals whether to show "-" or the actual percentage
- `DashboardAnalytics` is the single prop type passed from server component to client dashboard
- No `as` casts anywhere -- these types flow from zod-parsed data
